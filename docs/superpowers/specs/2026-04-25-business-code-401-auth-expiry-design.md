# 业务 code 401 登录失效处理设计

## 背景

后端重启后，旧 token 会失效。当前 `/api/admin/auth/me` 返回：

```json
{ "code": 401, "message": "未登录或登录已失效", "data": null }
```

该响应的 HTTP 状态码是 `200 OK`。现有前端只在 HTTP 状态码为 `401` 时触发重新认证逻辑，因此不会清理本地旧 token、用户信息和权限码，也不会跳回登录页，页面会停留在失效状态。

## 目标

- 当前端收到业务响应 `code === 401` 时，统一按登录失效处理。
- 清理本地登录态，包括 token、用户信息、权限码和动态路由状态。
- 跳转到登录页，并保留当前页面作为登录后的 `redirect`。
- 不调用退出登录接口，避免使用失效 token 再请求后端。
- 保持改动范围在 `apps/web-ele` 请求入口内，不修改 `packages/` 通用请求库。

## 非目标

- 不改变后端响应格式。
- 不启用 refresh token。
- 不重构通用请求库。
- 不调整业务页面、菜单路由、权限 meta 或登录表单。

## 方案

在 `apps/web-ele/src/api/request.ts` 增加 myjob 业务响应鉴权判断。该逻辑放在 `defaultResponseInterceptor` 之前，先读取后端原始响应体，识别 HTTP 2xx 里的业务失败码：

- 当响应体 `code === 401` 时，认为当前登录态已失效。
- 调用 `authStore.logout(true, false)` 执行本地退登。
- `notifyServer` 固定为 `false`，避免旧 token 触发退出接口递归或额外错误。
- 继续抛出带原始响应的错误，让后续错误提示链路仍能展示后端文案。

## 数据流

1. 页面或路由守卫请求接口，例如 `/admin/auth/me`。
2. 后端返回 HTTP `200`，响应体 `code` 为 `401`。
3. 请求入口识别业务登录失效。
4. 前端调用本地退登逻辑。
5. `logout()` 清理所有 Pinia store，重置动态路由。
6. 路由跳转到 `/auth/login`，并带上当前地址作为 `redirect`。

## 错误处理

登录失效处理应具备幂等性。多接口并发返回 `code === 401` 时，最终状态仍然是清空登录态并停留在登录页。实现时可以通过已有路由状态或轻量标记避免重复跳转和重复消息。

后端错误文案优先使用 `message` 字段，其次兼容 `msg`、`error`。这与现有错误提示提取逻辑保持一致。

## 测试设计

优先补纯函数测试，避免把请求拦截器和路由实例耦合进单测：

- 新增或扩展鉴权失败分类函数，覆盖 `code === 401` 的业务登录失效场景。
- 保留已有 `403` 账号禁用、用户组禁用、普通无权限分类测试。
- 如实现中新增业务响应判断函数，单测覆盖 `HTTP 200 + code 401`、`HTTP 401`、普通业务错误三类输入。

## 涉及文件

- 修改 `apps/web-ele/src/api/request.ts`：增加业务 code 401 的登录失效处理。
- 修改 `apps/web-ele/src/store/auth-session.ts`：如需复用判断，可补充小型纯函数。
- 修改 `apps/web-ele/src/store/auth-session.test.ts`：补充登录失效分类测试。

## 边界与风险

- 如果后端未来改为 HTTP `401`，现有 HTTP 401 拦截器仍然生效。
- 如果某些接口使用 `code: 401` 表达非登录态含义，会被统一退登；根据当前后端返回样例，该风险可接受。
- 如果多请求同时失效，可能出现重复错误提示；实现时应尽量只触发一次退登跳转。
